# Flow Guru Integration Architecture

Flow Guru will keep the existing memory-first chat loop, but the assistant send flow will become a small orchestration pipeline rather than a single LLM completion. Each authenticated message will follow five steps. First, the user message is persisted exactly as it is today. Second, a lightweight planner prompt produces structured JSON describing whether the user is asking for an actionable operation, which integration should handle it, and the normalized parameters needed to execute it. Third, the server executes the selected integration tool in the background. Fourth, the assistant generates the user-facing reply from both stored memory and the tool result. Fifth, memory extraction still runs after the reply so newly revealed preferences or routines continue to accumulate.

The orchestration layer should remain provider-agnostic. A typed action envelope can cover `calendar.create_event`, `calendar.list_events`, `music.play`, `route.get`, `weather.get`, and `news.get`. The planner should also support a `none` action so ordinary conversational messages still take the normal assistant path. This keeps Flow Guru natural for prompts like “play my workout music” or “what’s traffic like to the office” without exposing any manual API selection in the interface.

The first implementation milestone should prioritize the integrations that can run immediately without new credentials. Route and traffic lookups should use the existing server-side Google Maps proxy helper because it already supports geocoding and directions requests. Weather should use Open-Meteo because it accepts coordinate-based requests without user-specific OAuth. Personalized news should use an immediately available news source with structured responses and no key requirement, then map the user’s saved interests or recent topics into a small issue or query selection step before summarization.

Google Calendar and Spotify should be prepared in parallel as deferred account-linked providers. Their runtime handlers should exist behind connection checks so the assistant can respond gracefully when a provider has not yet been linked. The server should therefore gain a provider-connections model that stores the provider name, OAuth subject or external account identifier, encrypted or protected token fields, expiry metadata, granted scopes, and last refresh timestamp. The assistant can then say that a service is ready to connect instead of failing silently whenever the user asks for a calendar or music action before credentials are configured.

The natural-language user experience should stay minimal. The chat message response should include a concise assistant reply plus a structured `actionResult` payload for the frontend. That payload can render a calm inline result card inside the same chat stream, such as a route summary, a weather snapshot, a short news digest, or later a confirmation that an event was created or Spotify playback was triggered. This preserves the existing single-column chat-only design while still making actions feel tangible and trustworthy.

The database changes for the next implementation step should stay small and focused. One new table for provider connections is sufficient for the current roadmap, because route, weather, and news do not require user token storage. Google Calendar and Spotify can attach to that table later once the secrets are supplied. That lets us ship useful action behavior now without blocking on deferred OAuth setup.
